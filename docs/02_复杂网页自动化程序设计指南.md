# 02 — 复杂网页自动化程序设计指南

## 1. 明确目标

开始编写前需回答的问题：

- **操作哪个网站？** 每个网站结构不同，策略也不同
- **要完成什么任务？** 比如登录、填表、爬取数据、下单、发帖等
- **是否需要登录？** 登录涉及 Cookie/Session 处理
- **是否有反爬机制？** 比如验证码、IP 限制、行为检测

## 2. 架构设计思路

对于复杂场景，不是一份脚本从头到尾，而是分模块组织：

```
配置文件 (账号、URL、参数)
    ↓
初始化模块 (启动浏览器、加载配置)
    ↓
登录模块 (处理登录、Cookie 持久化)
    ↓
业务模块 (核心操作流程)
    ↓
异常处理 (超时重试、截图记录、日志)
```

## 3. 复杂场景常见问题

| 问题 | 解决思路 |
|------|---------|
| 页面元素动态加载 | 用 `waitForSelector` 等待元素出现 |
| 页面跳转/SPA 路由变化 | 等待网络请求完成 `waitForLoadState` |
| 弹窗/广告/通知 | 统一处理关闭逻辑 |
| 验证码 | Playwright 不擅长破解，建议手动跳过或接入打码服务 |
| 登录态丢失 | 保存 Cookie/LocalStorage，下次恢复 |
| 操作太快被检测 | 加入随机延迟 `setTimeout` + 随机数 |
| 多标签页 | 用 `browserContext.pages()` 管理 |

## 4. Playwright 的优势

- **自动等待元素** — 不必手动 sleep，`page.click()` 会等到元素可点击
- **支持多页面、多上下文** — `browser.newPage()` 同时操作多个标签页
- **网络拦截能力** — 可以 mock、拦截、修改 API 请求和响应
- **Cookie/Storage 持久化** — `browserContext.storageState()` 保存登录态
- **注入 JavaScript** — `page.evaluate()` 直接在页面内执行自定义脚本

## 5. 下一步

在开始编码前，确认以下信息：

1. **目标网站是什么？**
2. **要实现的具体操作流程？**（如：打开页面 → 登录 → 填表单 → 提交 → 截图）
3. **操作频率？** 一次性脚本，还是定期运行？
4. **有没有敏感信息？** 如账号密码

根据以上信息制定具体方案，再进行编码。
