# 03 — 项目目录架构设计

## 推荐目录结构

```
node_plawright_test/
├── package.json
├── playwright.config.ts           # 测试配置
├── .env                           # 环境变量（账号密码等，不要提交git）
├── .gitignore
│
├── src/                           # 自动化源码
│   ├── core/                      # 核心基础设施
│   │   ├── browser.ts             # 浏览器启动/关闭管理
│   │   ├── config.ts              # 配置加载（读取.env）
│   │   └── logger.ts              # 日志输出
│   │
│   ├── modules/                   # 业务功能模块（按场景拆分）
│   │   ├── login.ts               # 登录模块
│   │   ├── scraper.ts             # 数据采集模块
│   │   ├── form.ts                # 表单填写模块
│   │   └── ...                    # 每个场景一个文件
│   │
│   ├── utils/                     # 工具函数
│   │   ├── wait.ts                # 随机延迟、重试逻辑
│   │   ├── screenshot.ts          # 截图保存
│   │   └── storage.ts             # Cookie/Session 持久化
│   │
│   └── main.ts                    # 主入口（串联所有模块）
│
├── tests/                         # Playwright 测试（和 src 隔离）
│   └── example.spec.ts
│
├── output/                        # 运行产物（截图、数据导出等）
│   └── .gitignore
│
├── docs/                          # 文档
│   ├── 01_运行自动化操作的程序内容.md
│   └── 02_复杂网页自动化程序设计指南.md
│
└── COMMANDS.md                    # 命令参考
```

## 设计原则

| 原则 | 说明 |
|------|------|
| `src/core/` | 只放底层通用逻辑，不写具体业务 |
| `src/modules/` | 每个文件对应一个独立业务场景，互不影响 |
| `src/utils/` | 纯函数，可被任何模块复用 |
| `output/` | 运行产物和代码分离，不提交 git |
| `.env` | 敏感信息不写进代码 |

## 和现有结构的关系

- `amazon_test/` — 保持不变，这是测试模式下的测试用例
- `automation_demo.ts` — 保留作为参考，后续正式项目写进 `src/`

## 优势

- **扩展性强** — 新增一个业务场景只需在 `src/modules/` 下加一个文件，不需要改核心代码
- **职责清晰** — 核心层、业务层、工具层分离
- **可维护** — 每个模块独立开发、调试、替换
