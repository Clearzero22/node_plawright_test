# Playwright 爬虫接入工作流编辑器 — 经验总结

> 日期：2026-04-28
> 涉及项目：quick-test（爬虫）、workflow-editor（前端）

---

## 1. 架构设计

### 三层解耦架构

```
工作流编辑器 (React/Vite :5173)
  │
  │  ConfigPanel 根据 configSchema 渲染节点配置
  │  WorkflowEngine 串联执行节点、传递 config
  │  fetch /api/crawl/* (Vite proxy 转发)
  │
  ▼
API 服务器 (Hono :3456)
  │
  │  CrawlerService 编排生命周期
  │  BaseCrawler 模板方法模式 (navigate → extract → clean)
  │  DatabaseService 数据持久化
  │
  ▼
PostgreSQL (:5432) / 本地文件系统
```

**核心原则：任何一层都可以独立替换。** 爬虫不依赖前端，前端不依赖爬虫，API 是唯一桥梁。

### 数据流路径

```
configSchema 定义
  → useWorkflowState 初始化默认值到 nodeConfigs
    → ConfigPanel 渲染表单、用户编辑
      → setNodeConfig 写入 state
        → engine.execute(workflowNodes, callbacks, 0, nodeConfigsRef.current)
          → executor.execute({ config: nodeConfigs[nodeId] })
            → fetch('/api/crawl/gigab2b', { body: { url: config.productUrl } })
```

### 关键文件

| 层 | 文件 | 职责 |
|----|------|------|
| 前端 | `workflow-editor/src/plugins/index.ts` | 定义插件的 configSchema 和 execute |
| 前端 | `workflow-editor/src/engine/WorkflowEngine.ts` | 串联执行、传递 config |
| 前端 | `workflow-editor/src/hooks/useWorkflowState.ts` | 管理 nodeConfigs 状态 |
| 前端 | `workflow-editor/src/components/ConfigPanel.tsx` | 动态渲染节点配置表单 |
| API | `quick-test/src/api-server.ts` | REST 端点 |
| 服务 | `quick-test/src/services/crawler-service.ts` | 爬虫编排 |
| 爬虫 | `quick-test/src/crawlers/base-crawler.ts` | 模板方法基类 |
| 数据 | `quick-test/src/core/database-service.ts` | PostgreSQL 数据访问 |

## 2. 踩坑记录

### 2.1 `page.evaluate` 中不能用箭头函数

**错误信息：** `ReferenceError: __name is not defined`

**原因：** tsx 编译器将箭头函数转换为带 `__name` helper 的形式，注入到浏览器上下文后该函数不存在。

```typescript
// ❌ 错误 — tsx 编译后 __name 未定义
await page.evaluate(() => { ... });

// ✅ 正确 — function 声明不受影响
await page.evaluate(function () { ... });
```

**规则：Playwright `page.evaluate` 内部必须使用 `function` 声明。**

---

### 2.2 PostgreSQL 外键约束顺序

**错误信息：** `insert or update on table "raw_data" violates foreign key constraint`

**原因：** 子表（raw_data）引用父表（crawler_runs），但先执行了子表插入。

```typescript
// ❌ 错误顺序
await db.insertRaw(...);   // 失败：crawler_runs 中还没有这条记录
await db.insertRun(...);

// ✅ 正确顺序
await db.insertRun(...);   // 先插入父表
await db.insertRaw(...);   // 再插入子表
```

**规则：有外键依赖时，先插入父表再插入子表。**

---

### 2.3 PostgreSQL TEXT[] 数组类型

**错误信息：** `malformed array literal`

**原因：** 对 `TEXT[]` 列传了 `JSON.stringify(images)`，pg 驱动不认识这种格式。

```typescript
// ❌ 错误 — pg 驱动不会自动解析 JSON 字符串为 PostgreSQL 数组
await db.query('INSERT INTO ... VALUES ($1)', [JSON.stringify(images)]);

// ✅ 正确 — 直接传 JS 数组，pg 驱动会自动转换为 PostgreSQL 数组格式
await db.query('INSERT INTO ... VALUES ($1)', [images]);
```

**规则：pg 驱动直接传 JS 数组处理 PostgreSQL 数组类型，不要 JSON.stringify。**

---

### 2.4 数据库时区

**现象：** 存储的时间显示 UTC 而非北京时间。

**修复：**
```sql
ALTER DATABASE crawler_db SET timezone TO 'Asia/Shanghai';
```

或在连接字符串中指定：
```
postgresql://user:pass@localhost:5432/crawler_db?timezone=Asia/Shanghai
```

### 2.5 WorkflowEngine config 始终为空（核心 Bug）

**错误信息：** `缺少必填字段: url`

**原因：** `WorkflowEngine.execute()` 硬编码 `config: {}`，从未读取用户配置。

```typescript
// ❌ 原始代码
const output = await executor.execute({
  nodeId: node.id,
  config: {},  // 永远为空
  input: previousOutput ?? {},
  ...
});

// ✅ 修复后
async execute(
  nodes: EngineNode[],
  callbacks: EngineCallbacks,
  startIndex = 0,
  nodeConfigs: Record<string, Record<string, unknown>> = {},
): Promise<void> {
  ...
  const output = await executor.execute({
    nodeId: node.id,
    config: nodeConfigs[node.id] ?? {},
    input: previousOutput ?? {},
    ...
  });
}
```

**修复链路：**
1. `useWorkflowState` 添加 `nodeConfigs` state + `setNodeConfig` callback
2. `ConfigPanel` 根据 `configSchema` 动态渲染表单并绑定 state
3. 执行时 `engine.execute(workflowNodes, callbacks, 0, nodeConfigsRef.current)`
4. Engine 查找 `nodeConfigs[node.id]` 传给 executor

### 2.6 React 默认值陷阱（关键经验）

**现象：** ConfigPanel 显示了默认值（如 productUrl），但执行时 config 仍为空。

**原因：** UI 的 `?? field.default` 只影响渲染显示，`setNodeConfig` 只在 `onChange` 时触发。用户没有手动编辑过字段 = state 中根本没有数据。

```typescript
// ❌ 仅渲染时 fallback，state 中没有数据
const currentValue = nodeConfigs[nodeId]?.[key] ?? field.default ?? '';
// 用户没改过 → nodeConfigs[nodeId] 为 {} → currentValue 显示 default 但 state 未变

// ✅ 初始化时就预填 state
function buildDefaultConfigs(): Record<string, Record<string, unknown>> {
  const configs: Record<string, Record<string, unknown>> = {};
  for (const plugin of pluginRegistry.getAll()) {
    const schema = plugin.executor.configSchema;
    for (const [key, field] of Object.entries(schema)) {
      if (field.default !== undefined) {
        configs[plugin.id] ??= {};
        configs[plugin.id][key] = field.default;
      }
    }
  }
  return configs;
}

const defaultConfigs = buildDefaultConfigs();
const nodeConfigsRef = useRef(defaultConfigs);
const [nodeConfigs, setNodeConfigsState] = useState(defaultConfigs);
```

**规则：组件渲染的 fallback 默认值不影响 state。需要持久化传递的值，必须在 state 初始化时显式写入。**

## 3. 核心设计原则

### configSchema 驱动 UI

插件只需定义 `configSchema`，ConfigPanel 自动渲染表单，Engine 自动传值。新增一个节点类型只需要 30 行插件代码，零前端改动。

```typescript
// plugins/index.ts — 定义插件
configSchema: {
  productUrl: { type: 'string', label: '商品链接', required: true, default: '...' },
  headless: { type: 'boolean', label: '无头模式', default: true },
}
```

### 三层解耦

- **爬虫层** 独立运行，可以通过 CLI / API / 按钮触发
- **API 层** 纯 REST，不关心前端实现
- **前端层** 通过 fetch 调用 API，不关心爬虫实现

### 先读后写

改代码前先完整读懂数据流（UI state → engine → executor → API → service → crawler → db），不要猜。本次 config 为空的 bug 就是读完整个链路才定位到的。

## 4. 排查方法论

遇到问题时按以下顺序排查：

```
1. 读错误信息 → 精确定位报错位置
2. 追踪数据流 → 从 UI 到 API 到数据库，逐层检查数据是否正确传递
3. 对比差异 → 找到数据断裂点（哪一层丢失/篡改了数据）
4. 定位根因 → 针对性修复，而非绕过
```

本次 "缺少必填字段: url" 的排查路径：
```
API 报错 "缺少 url"
  → 读 api-server.ts：期望 body.url
    → 读 plugin execute：发送 { url: config.productUrl }
      → 读 WorkflowEngine：发现 config: {} 硬编码
        → 读 useWorkflowState：发现没有 nodeConfigs 机制
          → 根因定位，实施修复
```
