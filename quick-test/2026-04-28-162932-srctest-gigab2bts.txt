⏺ 这个脚本从 GigaB2B 网站提取以下数据：

  1. 基础信息
  - URL — 当前产品页面地址
  - 标题 — <h1> 或带有 title 类的元素
  - 价格 — .price / .product-price 等元素
  - 描述 — .product-description /
  .short-description（截取前200字符，备用方案用标题代替）

  2. 图片链接
  - 从所有 <img> 标签中提取，过滤条件：src 包含 image、product 或 b2bfiles
  的链接，排除小图标、logo、国旗图标
  - 从图片库容器（.el-image.image-item 等）中补充提取

  3. 规格参数（多种策略）
  - Product Info 标签页：从 .items 容器中提取 <span> 键值对
  - Specification 标签页：点击展开后，依次尝试4种抓取方式：
    a. 表格行（<tr> → <td>）
    b. 描述列表（<dl> → <dt> + <dd>）
    c. 键值对容器（label + value 配对）
    d. 正则匹配常见参数：Dimensions、Size、Weight、Material、Color、Package

  如果规格抓取失败，还会将页面正文前 5000 字符保存到
  output/gigab2b-page-content.txt 供人工分析。

  保存的输出文件：
  - output/gigab2b-product-data.json — 结构化 JSON
  - output/gigab2b-image-links.txt — 图片链接列表
  - output/gigab2b-product-page.png、gigab2b-image-viewer.png、gigab2b-final.png

  ---
  根据你的需求，我整理了 3种方案：

  ---
  方案 A：分层本地暂存 → 统一入库（推荐）

  核心思路：爬虫只负责抓取和初步解析，数据按层级存储，通过一个统一的
  CrawlerContext 管理整个生命周期。

    Crawler
       │
       ▼
    ┌─────────────────────┐
    │  Context 层          │  ← 记录：本次运行ID、来源、开始时间、状态
    │  (SQLite DB)         │
    └─────────────────────┘
       │
       ├── Raw Layer       │  原始 HTML / API 响应体（用于回放调试）
       ├── Staging Layer   │  解析后的结构化 JSON（清洗前）
       └── Clean Layer     │  清洗/规范化后的数据（入库就绪）

  Local 存储结构：
  output/
  ├── db/                          # SQLite 运行元数据库
  │   └── crawler.db               # 记录每次运行、状态、统计
  ├── <source>/<run-id>/
  │   ├── raw/                     # 原始 HTML
  │   ├── staging/                 # 解析后的中间 JSON
  │   ├── clean/                   # 清洗后的数据
  │   └── screenshots/             # 截图
  ├── 2_gigab2b/2026-04-28_abc123/
  │   ├── raw/product-page.html
  │   ├── staging/product-data.json
  │   ├── clean/product-ready.sql
  │   └── screenshots/final.png

  并行写入 DB：
  - PostgreSQL → 核心业务数据（产品、价格、规格）
  - MongoDB → 非结构化数据（评论文本、图片链接列表、长描述）
  - SQLite → 本地运行日志 + 暂存队列（离线模式下积压，连上 PG 后回写）

  优点：每层解耦，抓取不依赖数据库可用性，可重跑某层而不重抓
  缺点：需要额外维护一个 Context 中间层

  ---
  方案 B：直接入库模式

  前端就是现在这种每个脚本自己定义接口 → 解析后直接往目标 DB
  写。改进点在于引入一个共享的 DatabaseService 抽象。

  Crawler → DatabaseService → PostgreSQL / MongoDB / SQLite

  优点：结构简单，少一层文件操作
  缺点：数据库挂了就白跑，中间数据不可回溯，调试困难

  ---
  方案 C：事件驱动管道

  每个爬虫步骤发出事件，通过事件总线流转：

  Crawler → emit("page:loaded") → RawHandler → emit("data:parsed") →
  CleanHandler → emit("data:ready") → DBWriter

  优点：扩展性强，容易加新消费者（比如实时通知、数据校验）
  缺点：对你目前的规模来说过度设计了，维护成本高

  ---
  我的推荐：方案 A。理由：
  1. 你同时用 PG / SQLite / MongoDB，层次化存储能自然映射到不同数据库
  2. 中间数据可回溯 — 如果入库后发现数据有问题，可以直接看 staging 层对比
  3. 不依赖目标 DB 可用性 — 离线也能抓，连上网再回写
  4. 适合手动 + 定时两种运行模式

  这个方向你觉得合理吗？如果方向 OK，我接着设计具体的分层数据结构。
